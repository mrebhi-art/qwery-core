# Instrumentation: Tracing + Evals (Agent Factory SDK)

This document explains the eval instrumentation setup and how to run it for the first time.

## Scope

- Tracing is implemented in the agent layer and emits structured steps for LLM, tool, retrieval, and custom events.
- Evals run via the tracing SDK and report metrics, token usage, latency, and tool calls.
- The semantic layer is used when benchmarks execute data-agent workflows that require ontology and query execution.

## Where Tracing Happens

- Agent-level tracing interface: `src/agents/tracing.ts`
  - Step types: `llm_call`, `tool_call`, `retrieval`, `reasoning`, `custom`
  - Each step includes input, output, latency, and optional token usage
- Eval harness uses the tracing SDK to collect traces and aggregate metrics.

## Where Eval Suites Live

- Evals entrypoint: `evals-regression/`
- Benchmarks: `evals-regression/suites/benchmarks/`
- Shared harness helpers: `evals-regression/suites/_shared/`

See the base evals README for a full suite list:
- `evals-regression/README.md`

## First-Time Setup

1) Install dependencies from repo root:

```bash
pnpm install
```

2) Start the eval backend (required):

```bash
# qwery-eval backend
# default: http://localhost:4097

# qwery-eval-ui (optional but recommended)
# default: http://localhost:5090
```

3) Configure required environment variables:

```bash
# Eval service
EVAL_BASE_URL=http://localhost:4097
EVAL_UI_BASE_URL=http://localhost:5090

# Model selection
EVAL_MODEL=ollama-cloud/minimax-m2.7

# Optional run labeling
AGENT_VERSION=local
EVAL_PROJECT_ID=semantic-qwery-core
```

4) Prepare BIRD dataset access (required for benchmark suites):

Option A: Local JSON dataset
```bash
BIRD_DATASET_ROOT=C:\path\to\bird-mini-dev
# expects: mini_dev_sqlite.json, mini_dev_mysql.json, mini_dev_postgresql.json
```

Option B: Single JSON file
```bash
BIRD_DATASET_JSON=C:\path\to\mini_dev_sqlite.json
```

## First-Time Run: BIRD California Schools

From `packages/agent-factory-sdk`:

1) Build ontology (required once per dataset):

```bash
bun run eval:benchmark:bird:build-ontology:schools
```

2) Run the benchmark:

```bash
bun run eval:benchmark:bird:schools
```

If you run from repo root instead:

```bash
pnpm --filter @qwery/agent-factory-sdk eval:benchmark:bird:build-ontology:schools
pnpm --filter @qwery/agent-factory-sdk eval:benchmark:bird:schools
```

## Outputs and Artifacts

- Eval results:
  - `evals-regression/reports/`
- BIRD cache pages:
  - `evals-regression/reports/bird-cache/`
- Temporary benchmark databases:
  - `.tmp/internal-agent/`
- Local storage:
  - `qwery.db/` (root-level)

## Do Not Commit Generated Data

The following are generated by eval runs and should not be pushed:

- `packages/agent-factory-sdk/evals-regression/reports/`
- `packages/agent-factory-sdk/evals-regression/reports/bird-cache/`
- `packages/agent-factory-sdk/.tmp/`
- `qwery.db/`
- Any locally downloaded BIRD datasets

If needed, add these paths to `.gitignore` in your branch.

## Troubleshooting

- If datasets do not load on first run:
  - Set `BIRD_DATASET_ROOT` or `BIRD_DATASET_JSON`
  - Ensure HuggingFace is reachable for metadata fetch
- If ontology is missing:
  - Re-run `eval:benchmark:bird:build-ontology:schools`
- If evals do not report:
  - Confirm `EVAL_BASE_URL` is reachable
  - Check backend logs for auth or model errors
